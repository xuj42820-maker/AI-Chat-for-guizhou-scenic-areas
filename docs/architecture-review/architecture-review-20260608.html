Problem

main.py is a 473-line flat file — page rendering, attraction CRUD, route queries, favorites, statistics, database access, and app init all share one module. The interface is 20+ routes; the implementation is the same 20+ routes. Shallow: interface ≈ implementation.

Solution

Split into focused Flask Blueprints: routes/pages, routes/attractions, routes/routes, routes/favorites, routes/stats. Move get_db() to a shared db.py. main.py becomes a 30-line app factory.

Wins

  • Locality: each feature's routes, queries, and error handling live in one module
  • Leverage: each Blueprint has a focused interface (3–7 routes, not 20+)
  • Test surface shrinks: test attraction queries without loading page routes
  • Delete test: remove any one Blueprint — complexity stays local, doesn't scatter

Problem

Three modules independently resolve db_path, open connections, set row_factory, and close. database.py's four functions each do the same open-work-close cycle. No seam: swapping SQLite for tests means editing every call site.

Solution

Extract db.py with get_db(), close_db(), and a with_db(query_fn) context manager. All modules call through one interface. Two adapters: real SQLite and in-memory for tests.

Wins

  • Locality: connection lifecycle bugs concentrate in one module
  • Leverage: one interface, 4 call sites
  • Two adapters justify the seam: SQLite in prod, :memory: in tests
  • Delete test: remove db.py — every module re-implements open/close/row_factory

Problem

RAGQuestionAnswer loads data, builds TF-IDF indexes, performs search, detects intent, AND generates answers. The _generate_answer method alone is a 180-line if-elif chain — adding a new intent means editing three methods in the same class.

Solution

Extract SearchIndex (TF-IDF build + cosine search), IntentClassifier (keyword matching + city extraction), and AnswerGenerator (template per intent). RAGQuestionAnswer becomes a thin facade composing the three.

Wins

  • Locality: each concern (search, intent, generation) changes independently
  • Interface is the test surface: test intent detection without building TF-IDF index
  • Adding a new intent = one new entry in IntentClassifier + one template in AnswerGenerator
  • SearchIndex can be swapped (BM25, embedding-based) without touching intent or generation

Interface (what callers must know)

Implementation (hidden behind the interface)

create_tables()
import_attractions()
import_images()
import_guides()
import_routes()
connection lifecycle
path resolution
transaction management

Interface shrinks; implementation absorbs the details

Problem

database.py's four functions are scripts disguised as a module. Each independently resolves the db_path, opens a connection, does work, and closes. The interface is "call these 5 functions in order" — no simpler than the implementation. Shallow.

Solution

Wrap in a Database class. Constructor takes db_path. One public method: seed(json_path) — internally calls create_tables + all imports in a single transaction. Callers learn one method; everything else is implementation.

Wins

  • Leverage: callers invoke one method, not five in sequence
  • Locality: import ordering, transaction boundaries, error handling — all inside
  • Testable: pass ":memory:" as db_path, seed() a small JSON fixture
  • Combines naturally with Candidate 2's db.py seam

Extract stats.py with get_overview(). Single region list, single query strategy. Both endpoints call through one interface.

Wins

  • Consistency: admin and public endpoints return identical structure
  • Low effort: extract ~50 lines, replace two inline implementations

Top Recommendation

1st

Unify database access behind a repository seam

Highest leverage: fixes the scattered-connection problem across all 4 modules, creates a real seam with two adapters (SQLite / :memory:), and unblocks every other refactor. Tackle this first — the others become easier once db access is clean.

2nd

Split the main.py god module

Once db.py exists, splitting main.py into Blueprints is mechanical: move routes, swap inline get_db() calls for the new seam. Each Blueprint becomes independently testable.

3rd

Decompose rag_qa.py

Independent of the db refactor. Splitting search / intent / generation lets you test intent detection without building TF-IDF indexes, and makes adding new intents a one-file change.
