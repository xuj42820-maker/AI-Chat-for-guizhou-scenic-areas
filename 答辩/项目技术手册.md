## 一、项目是干什么的？

一句话：**你问问题，它给答案。**

比如你问"黄果树瀑布门票多少钱"，系统从数据库里找到黄果树瀑布的信息，组织成一段完整的话回答你，还告诉你答案是从哪来的。

它不是百度搜索（给你一堆链接），也不是ChatGPT（可能胡说八道）。它是**基于你自己的数据**来回答的，所以答案是靠谱的。

## 二、整体架构（四层）

整个系统分成四层，从上到下：

```
┌─────────────────────────────────────────┐
│  前端层（用户看到的界面）                    │
│  HTML + CSS + JavaScript + Leaflet地图    │
├─────────────────────────────────────────┤
│  后端层（处理请求的服务器）                  │
│  Python + Flask                          │
├─────────────────────────────────────────┤
│  AI核心层（智能问答的大脑）                  │
│  意图识别 + TF-IDF检索 + 回答生成(RAG)     │
├─────────────────────────────────────────┤
│  数据层（存数据的地方）                     │
│  SQLite数据库 + JSON文件                   │
└─────────────────────────────────────────┘
```

### 每层干什么？

**前端层**：就是用户打开网页看到的东西。包括首页（搜索框、Banner轮播）、景点地图（Leaflet地图标注117个景点）、聊天界面（AI问答）、管理后台（增删改查）、数据可视化大屏。

**后端层**：用Python Flask搭建的服务器。它的工作是：接收前端发来的请求 → 调用AI核心处理 → 把结果返回给前端。Flask通过Blueprint机制把不同功能的路由分到不同文件里，代码结构清晰。

**AI核心层**：这是最关键的部分。它做三件事：
1. 意图识别——判断你在问什么类型的问题
2. TF-IDF检索——在117个景点里找到和你问题最相关的
3. 回答生成（RAG）——把检索到的资料整理成一段完整的回答

**数据层**：SQLite数据库存着所有结构化数据（景点、图片、攻略、路线、收藏）。JSON文件存着一些辅助数据（餐厅、旅行贴士）。

## 五、数据库设计（6张表）

### 表结构

| 表名 | 作用 | 关键字段 |
|------|------|---------|
| **attractions** | 景点基本信息（主表） | id, name, address, category, ticket_price, opening_hours, description, rating, latitude, longitude |
| **images** | 景点图片 | id, attraction_id(外键), image_url, image_type, is_primary |
| **guides** | 景点攻略 | id, attraction_id(外键), title, content, guide_type |
| **routes** | 旅游路线 | id, name, description, duration, difficulty, attractions_list |
| **favorites** | 用户收藏 | id, user_id, attraction_id(外键), created_at |

### 表之间的关系

```
attractions (主表)
    ├── 1:N ── images (一个景点有多张图片)
    ├── 1:N ── guides (一个景点有多篇攻略)
    └── 1:N ── favorites (一个景点被多人收藏)

routes (独立表，通过attractions_list文本记录包含的景点)
```

### 为什么这样设计？

- **attractions是核心**：所有其他表都和它关联
- **图片和攻略分开存**：一个景点可以有很多图片和很多攻略，用外键关联比塞在一张表里更灵活
- **routes独立**：路线是组合概念，用文本记录包含哪些景点，简单够用
- **遵循三大范式**：没有字段重复，没有传递依赖，外键保证一致性

## 六、AI问答系统详解

### 6.1 意图识别 (intent.py)

**干什么**：判断用户的问题属于哪一类。

**5种意图**：
1. 门票价格查询 —— "门票多少钱"、"票价多少"
2. 交通路线查询 —— "怎么去"、"坐什么车"
3. 开放时间查询 —— "几点开门"、"营业时间"
4. 景点推荐 —— "有什么好玩的"、"推荐一下"
5. 综合问答 —— 其他问题

**怎么实现**：通过关键词匹配。比如问题里包含"门票""票价""多少钱"→ 门票价格查询。

### 6.2 TF-IDF检索 (search.py)

**干什么**：在117个景点中找到和用户问题最相关的。

**分词方法**：不用jieba，用n-gram。把"黄果树瀑布"切成：黄果、果树、树瀑、瀑布（2-gram）+ 黄果树、果树瀑、树瀑布（3-gram）。不需要任何外部依赖。

**TF-IDF算法**：
- TF（词频）：一个词在当前文档中出现越多，越重要
- IDF（逆文档频率）：一个词在所有文档中出现越少，越独特
- TF-IDF = TF × IDF：既频繁又独特的词，分数最高

**检索过程**：
1. 对用户问题分词，计算每个词的TF-IDF分数
2. 对数据库中每个景点的文本也做同样的分词和计算
3. 用余弦相似度比较问题向量和每个景点向量的相似程度
4. 按相似度排序，返回Top-5

**额外加分**：如果景点名称直接包含用户问题中的关键词，额外加0.15分。

### 6.3 回答生成 (generator.py)

**干什么**：把检索到的景点信息组织成一段完整的回答。

**根据意图选择信息**：
- 门票查询 → 输出门票价格、优惠政策
- 交通查询 → 输出交通方式、地址
- 时间查询 → 输出开放时间、建议游玩时长
- 推荐查询 → 输出景点特色、评分
- 综合问答 → 输出所有信息

**输出格式**：Markdown格式，支持标题、列表、加粗等，前端用marked.js渲染。

## 七、前端交互详解

### 7.1 聊天系统 (chat.js)

**ChatSystem对象**的核心方法：
- `init()` — 初始化，绑定事件
- `send(text)` — 发送消息，调用API
- `addUserMessage(text)` — 显示用户消息气泡
- `addAIMessage(data)` — 显示AI回答（流式输出）
- `streamText(el, text)` — 逐字显示效果（模拟打字机）
- `showThinking()` — 显示"正在思考..."动画
- `buildAttractionCards(attractions)` — 生成景点推荐卡片
- `buildSuggestions(suggestions)` — 生成追问建议按钮

**流式输出**：不是真的流式传输，而是前端模拟的——把文字按段落分割，逐字显示，标点符号后停顿更长，模拟真人打字的感觉。

### 7.2 地图模块 (map.html + Leaflet)

- 使用Leaflet.js库
- 从 `/api/attractions` 获取所有景点数据
- 在地图上标注117个标记点
- 点击标记显示景点信息弹窗
- 侧边栏显示景点列表，支持按类别筛选
- 路线可视化：在地图上画出路线连线

### 7.3 管理后台 (admin/)

- Blueprint注册在 `admin.py`
- 支持景点的增删改查
- 支持路线的增删改查
- 支持攻略的查看和编辑
- 数据写入SQLite数据库

## 十、老师可能问的问题

### Q: 为什么用Flask不用Django？
A: Flask更轻量、灵活，适合中小型项目。我们的项目不需要Django那么多内置功能（用户认证、ORM等），Flask足够了。

### Q: 为什么不用jieba分词？
A: jieba需要额外安装，n-gram不需要任何外部依赖，部署更简单。对于我们的场景（景点文本检索），n-gram效果够用。

### Q: TF-IDF和embedding有什么区别？
A: TF-IDF是基于词频的统计方法，简单快速但不理解语义。embedding是把文字转成数字向量，能理解语义相似（比如"瀑布"和"水帘"）。我们用TF-IDF是因为它轻量、无需训练，对于景点文本检索场景效果足够。

### Q: RAG和直接问ChatGPT有什么区别？
A: ChatGPT的回答基于它训练时学到的知识，可能过时或编造。RAG基于我们自己的数据库回答，数据是真实的、可控的。

### Q: 数据库为什么用SQLite不用MySQL？
A: SQLite不需要单独安装数据库服务器，整个数据库就是一个文件，适合小型项目和本地部署。如果项目需要多用户并发访问，可以换成MySQL。

### Q: 这个项目有什么不足？
A: 缺少用户账号系统、不支持多轮对话记忆、景点数据有限、未实现语音交互。

### Q: 后续怎么改进？
A: 加用户系统、接大语言模型做多轮对话、爬更多景点数据、做移动端适配。
